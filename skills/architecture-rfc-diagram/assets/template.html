目标不是让 LLM “猜架构”,而是把 repo scan → 证据提取 → 架构归纳 → Mermaid 渲染 → 人工复核 做成固定流水线。 filesystem MCP 负责读代码,git/github MCP 负责补上下文,architecture-rfc-diagram skill 负责把结果稳定输出成本文档风格。

C · PROMPT CONTRACT

Claude Code 调用约定

sequenceDiagram
FIG. C 
sequenceDiagram
    autonumber
    participant U as user
    participant CC as Claude Code
    participant MCP as filesystem/git MCP
    participant SK as architecture skill
    participant OUT as HTML RFC

    U->>CC: Analyze this repo and draw architecture page
    CC->>MCP: list tree + read build/config/entrypoints
    MCP-->>CC: files + snippets + git context
    CC->>SK: normalize evidence into architecture model
    SK->>SK: classify modules / flows / boundaries
    SK-->>CC: Mermaid blocks + notes + risk matrix
    CC->>OUT: render single-file HTML
    OUT-->>U: reviewable architecture document

    Note over CC,SK: Any node without code evidence is marked inferred, not fact.
  • 推荐让 Claude Code 每次先产 architecture_model.json,再产 HTML;这样后续改图不会重新“脑补”。
  • 第一次输出不要追求完美,先把边界画出来;第二轮再让人补 owner / latency / fault domain / rollout。
D · DELIVERABLES

Skill 应该固定产出的章节

matrix
章节 内容 证据来源 可信度规则
TOPOLOGY 进程、服务、模块、外部依赖 entrypoint、config、compose/systemd 必须有路径证据
INTERFACE 接口、adapter、client、repository class/interface/proto/header 符号名可追溯
SEQUENCE 核心请求链路、启动链路、数据链路 caller/callee、topic、RPC、queue 至少两段调用证据
UNKNOWN 推断节点、缺失配置、需要人确认的风险 命名、目录结构、注释 必须显式标记 inferred
E · TRADEOFFS

为什么这么设计

3 picks · 1 line each
  1. 用 Skill 固化风格,而不是每次临场 prompt —— 架构图的稳定性来自模板、术语表、证据规则和输出 schema,不是一次性的“请画漂亮点”。
  2. 先产结构化模型,再产 Mermaid —— architecture_model.json 可以 review、diff、复用;HTML 只是最终渲染层。
  3. 把 unknown 画出来 —— 新项目最危险的不是图不完整,而是把推断画得像事实;红框/虚线能强迫 reviewer 补证据。
