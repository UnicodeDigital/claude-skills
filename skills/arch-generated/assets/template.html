ARCHITECTURE · pandora

Pandora 交易系统架构

OWNER trading-infra
STATUS Living Document

Pandora 是一套面向 10+ 加密交易所(Binance/Bybit/OKX/Gate/Bitget/Kucoin/MEXC/Hyperliquid/Aster/Lighter)的 C++ 量化交易系统,覆盖从实时行情、策略执行、订单/持仓/余额管理到交易所对接的完整链路。 策略开发者继承 StrategyBase:系统把实时行情(MD)、订单/持仓/余额(OMS)、历史回放(Sim)抹平成同一套虚函数回调,在回调里决策、调 sendOrder 下单; 每个交易所一个 Executor 把请求翻译成该所协议,经 Exchange::Agent + SessionBundle(自实现的 TCP/TLS/HTTP/HTTP2/WS 网络栈,底层 mbedtls + nghttp2)发往交易所。 所有回调都在主线程触发,策略逻辑天然单线程、无需加锁;跨线程边界用 SPSC 队列 + 内存池。

  • L1 · STRATEGY:策略层。继承 StrategyBase、override 回调、调发单/订阅接口;L2–L5 都在回调背后。
  • L2 · 行情源:实盘走 MD::Agent;回测时 MdLoader 取代 MD::Agent(读 Arrow/Zstd 历史数据),策略回调代码不变;Kafka/Redis 视策略需求开关。
  • L3 · OMS:订单/持仓/余额。按 vendor + market 各一个 OrderManager;每个交易所一个 Executor 子类,把"下单/撤单/查询"翻译成该所协议,协议/签名/限频差异全封在 Executor 内、策略无感。
  • L4 · 协议/网络:Exchange::Agent 负责多 vendor 的编解码,SessionBundle 是自实现的网络栈(TCP/TLS/HTTP/HTTP2/WS,底层 mbedtls + nghttp2)。
  • L5 · 交易所/历史数据:实盘连 10+ 交易所;回测读 Arrow/Zstd 历史数据。断线/重连由 SessionBundle 兜底(见 D)。
B · GETTING STARTED onboarding
  1. 继承 StrategyBase,override 关心的回调(行情 / 订单 / 账户)。
  2. init("config/strategy.cfg") 读 venue + 模块配置,subscribeMd(...) 订阅行情。
  3. run() 进事件循环——之后所有回调在主线程被触发,在回调里 getOrderManager(...).sendOrder(...)。
#include "strat/strategy_base.h"
#include "oms/order_manager.h"
using namespace strat;

class MyStrategy : public StrategyBase {
  // —— 行情回调:订阅了对应 topic 才会来 ——
  void onBookTicker(const MD::BookTicker& bt) override { /* BBO 更新 → 决策 */ }
  void onTrade(const MD::Trade& t) override {}

  // —— 订单回调 ——
  void onOrderAcked (const ResponseHeader& h, const OrderUpdate& o) override {}
  void onOrderFilled(const ResponseHeader& h, const OrderUpdate& o) override {}
  void onOrderRejected(const ResponseHeader& h, const ErrorMsg& e) override {}

  // —— 账户回调 ——
  void onBalanceUpdate(BalanceUpdate* r) override {}

  // —— 下单:在任意回调里调 ——
  void place(const Symbol& sym, Side side, Price px, Quantity qty) {
    RequestHeader h{};
    h.vendor = Exchange::Vendor::BINANCE;        // 选哪个所
    h.market = Exchange::Market::LINEAR;         // 现货 SPOT / U 本位 LINEAR ...
    h.account_type = AccountType::CROSSMARGIN;
    string_to_char_array("my_account", h.account_id);

    NewOrder o{};
    string_to_char_array(sym, o.symbol);         // "btc-usdt"
    o.price = px; o.qty = qty; o.side = side;
    o.type = OrderType::LIMIT; o.tif = OrderTif::GTC;
    // client_order_id 不赋值:OMS 启动后自动分配自增 cid;
    // 仅当需要自定义编号布局时,才显式 o.client_order_id = ...;

    getOrderManager(h.vendor, h.market).sendOrder(&h, &o);
    // 结果异步回到 onOrderAcked / onOrderRejected
  }
};

int main() {
  MyStrategy s;
  s.init("config/strategy.cfg");                 // venue + 模块
  s.subscribeMd(Exchange::Vendor::BINANCE, Exchange::Market::LINEAR,
                MD::TopicType::BookTicker, "btc-usdt");
  s.run();                                       // 事件循环 · 回调在主线程触发
}
  • 发单/撤单/查询:getOrderManager(vendor, market) 拿到对应 OrderManager,再调 sendOrder(&header, &order) / cancelOrder(client_order_id) / queryOpenOrders / queryBalance / queryPosition。本地还可 findOrder / getOpenOrders / hasInFlightOrder 查缓存。
  • 订阅行情:subscribeMd(vendor, market, MD::TopicType, symbol, settings)。TopicType:Trade · Orderbook · PartialOrderbook · MarkPrice · BookTicker · AggTrade · Kline。订阅了哪个 topic,对应 onXxx 才会被回调。
  • 定时器:setTimer(interval_s, handler, is_periodic, run_immediately) 注册周期/单次回调(轮询、对账、撤超时单常用)。
  • 完整的发单/撤单/查询用法参考 src/examples/oms_test.cpp。

▼ 可 override 的回调(按来源分组)

来源回调何时触发
行情 MD onTrade · onOrderbook · onPartialOrderbook · onBookTicker · onMarkPrice · onKline 订阅的 topic 有新数据(subscribeMd 后)
订单 OrderListener onOrderAcked · onOrderFilled · onOrderCanceled · onOrderExpired · onOrderRejected · onCancelRejected · onQueryOpenOrdersResp 下单/撤单的回报、成交推送、查单响应
账户 AccountListener onBalanceUpdate · onPositionUpdate · onQueryBalanceResp · onQueryPositionResp · onQueryAccountInfoResp 余额/持仓推送、对应 query 的响应
连接状态 onDataStreamSubscribe · onSubscribeDisconnected 数据流订阅成功 / 断连(对账时机,见 D)
通用错误 onUnifiedErrorResp 各类请求的统一错误兜底

  • 下单(C1):策略 sendOrder 后,请求经 OrderManager → OrderDispatcher → Executor → Agent 逐级转换发往交易所;回报沿 Agent → Executor → OrderDispatcher → OrderManager 返回,最终在主线程触发 onOrderAcked / onOrderFilled。撤单、查询走同一条链路,只是请求类型不同。
  • 订单/持仓/余额的主动推送(UserDataStream)走完全相同的回程,不是另一条独立 stream——只是事件类型不同,最终回到 onPositionUpdate / onBalanceUpdate。
  • 行情(C2):subscribeMd 后行情流持续推送,MD::Agent 解析归一化成 MD::* 结构,回到 onTrade / onBookTicker 等;与订单回调同在主线程事件循环。
  • 这是 happy path;断线 / 重连见 D。
D · STATE orders · positions · balance · sessions
状态 以谁为准 本地缓存 / 在线更新 断线后 / 恢复
挂单 交易所 OrderManager 内部 order cache;靠成交推送 + 自己 send/cancel 的回报更新 框架定时巡检:正常 5s 一轮查超时,断线后提速到 1s;策略可在 onSubscribeDisconnected 里主动 queryOpenOrders 对账
持仓 交易所 PositionManager 从 onOrderFilled 增量;持仓推送回填 启动时策略 initPositions;框架不自动对账,策略按需 queryPosition
余额 交易所 经 onBalanceUpdate 透传;框架自身不存余额视图 策略主动 queryBalance → onQueryBalanceResp
连接 SessionBundle 连接池 + 心跳 + 重连定时器 底层自动重连(默认开,1s 间隔),向上 emit Session_Disconnected / Session_Connected
  • 本地缓存可能比交易所慢几个事件,一切以交易所数据为准。 框架只对挂单做定时巡检,持仓/余额不自动对账、由策略按需 query。策略要能容忍"本地比交易所落后几个事件"的窗口(比如 onOrderFilled 还没到、但仓位实际已变)。
  • 断线时框架的动作:SessionBundle 自动重连;挂单巡检 5s→1s;透传 onSubscribeDisconnected(WS_STREAM 断时)。挂单不动 / 仓位不变 / 余额不重置。
  • 断线后策略需处理的:在 onSubscribeDisconnected 里决定是否 queryOpenOrders 对账、是否撤掉旧挂单、是否重发失败的单。重连恢复后框架回调 onDataStreamSubscribe、巡检恢复 5s。注意:断线期间交易所那边挂单可能仍有效(尤其 GTC),不主动对账会和本地分歧。
E · BOUNDARY 分工 + 取舍
  1. 框架负责:消息无丢失地传到 / 传回 + vendor 协议正确。 出向把请求按 vendor+market 翻译成交易所协议发出,入向把回报/推送无锁地送回回调;10+ 交易所的协议、签名、限频、断线重连差异全替策略抹平。策略拿到的永远是统一的 OrderUpdate / BalanceInfo / MD::* 结构。
  2. 策略负责:对账与风控。 本地缓存可能滞后、一切以交易所为准,框架不替策略做任何对账;pre-trade 风控、跨 vendor 仓位聚合、断线后对账策略(cancel-all / query-and-merge / 信任推送)都在策略层。这些需求各策略口径差异巨大,暂不在框架层抽象——好处是框架边界清晰,策略的风控规则变化不会触发框架回归。
  3. 断线只有一个权威源(SessionBundle)。 重连固化在最底层网络栈,上面所有层只通过 Session_Disconnected / Session_Connected 事件感知并各自反应——不会出现多层抢着重连的"重连风暴"。代价:每层(含策略)要显式实现"下层断了该怎么办"。
